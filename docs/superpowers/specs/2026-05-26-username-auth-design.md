# 用户名登录注册改造设计

## 背景

当前项目的登录注册体系基于邮箱或手机号账号，包含验证码发送、验证码注册、验证码登录、密码登录等多条链路。现阶段产品需求已经收敛，不再需要邮箱验证码、手机号登录，也不需要兼容现有老账号数据。

本次改造目标是将认证体系统一为用户名体系，减少链路复杂度，修正数据语义，并为后续功能迭代保留清晰边界。

## 目标

- 注册改为 `用户名 + 密码 + 重复密码`
- 登录改为 `用户名 + 密码`
- 删除邮箱验证码发送、验证码注册、验证码登录链路
- 保留 Turnstile 人机验证
- 保留 `nickname` 作为显示昵称，但不等于用户名
- 注册时由后端自动生成默认昵称，后续用户可自行修改
- 不兼容现有老账号数据，按新站处理

## 非目标

- 不支持手机号登录
- 不支持邮箱登录
- 不支持短信或邮件验证码
- 不支持用户名修改
- 不做找回密码流程
- 不保留旧认证接口的向后兼容行为

## 方案对比

### 方案 A：新增 `username` 作为唯一登录标识

- `users` 表新增 `username` 字段并加唯一约束
- 登录注册接口全面切换到 `username`
- JWT 身份载荷改为 `userId + username + role`
- 现有 `phone` 字段仅作为历史字段暂时保留，不再参与认证

优点：

- 字段语义正确
- 后续维护成本最低
- 认证链路最清晰

缺点：

- 需要同时改数据库、后端接口、前端页面和本地存储结构

### 方案 B：复用 `phone` 字段承载用户名

- 前端改成用户名表单
- 后端和数据库仍沿用 `phone` 字段

优点：

- 改动少

缺点：

- 语义错误
- 后续代码会持续混淆“用户名”和“手机号”
- 会拖累后续权限、资料、审计相关开发

### 方案 C：双字段并存但只暴露用户名登录

- 新增 `username`
- 旧体系结构仍保留较多兼容逻辑

优点：

- 对老数据更平滑

缺点：

- 本次需求明确不兼容老账号，这部分复杂度没有收益

### 最终选择

采用方案 A。

## 数据模型设计

### User 模型

当前 `User` 以 `phone` 为主身份字段。改造后：

- 新增 `username: STRING`
  - `allowNull: false`
  - `unique: true`
- `username` 规则：
  - 只允许字母、数字、下划线
  - 长度 4 到 20 位
  - 正则：`^[A-Za-z0-9_]{4,20}$`
- `nickname` 保留
  - 注册时由后端自动生成
  - 默认昵称不等于用户名
- `phone` 字段短期保留
  - 不再参与登录注册
  - 不作为 JWT 主身份字段

### 默认昵称生成

由后端统一生成，避免前后端各自实现规则。

建议格式：

- 前缀固定为 `用户`
- 后缀为 6 位随机大写字母或数字
- 示例：`用户A7K2Q9`

需要保证：

- 昵称允许重复
- 生成失败时可重试

## 接口设计

### 保留接口

#### `POST /api/auth/register`

入参：

- `username`
- `password`
- `captchaToken`

行为：

- 校验用户名格式
- 校验密码长度
- 校验 Turnstile
- 检查用户名唯一性
- 创建用户
- 自动生成默认昵称
- 返回 token 与用户信息

返回：

- `token`
- `username`
- `nickname`
- `role`

#### `POST /api/auth/login-password`

入参：

- `username`
- `password`
- `captchaToken`

行为：

- 校验用户名格式
- 校验 Turnstile
- 按 `username` 查询用户
- 校验密码
- 更新最后登录时间
- 返回 token 与用户信息

返回：

- `token`
- `username`
- `nickname`
- `role`

#### `GET /api/auth/verify`

保留，但基于新的 JWT 身份信息返回用户状态。

### 删除或停用接口

- `POST /api/auth/send-code`
- `POST /api/auth/login`（验证码登录）

处理方式：

- 前端不再调用
- 后端应明确停用，返回清晰错误，或直接删除注册逻辑入口

本次优先建议直接停用并删除其调用路径，避免保留半废弃逻辑。

## JWT 与鉴权设计

当前 JWT 主要写入 `phone` 与 `role`。改造后：

- `signUserToken(user)` 改为写入：
  - `userId`
  - `username`
  - `role`
- 鉴权解析用户时优先按 `userId` 查询
- `req.user` 中不再依赖 `phone` 作为唯一身份源

原因：

- `userId` 稳定且唯一
- `username` 可用于调试与接口返回
- 继续依赖 `phone` 会把旧语义延续到新体系中

## 前端设计

### 登录页

表单项：

- 用户名
- 密码
- Turnstile

行为：

- 用户名按统一正则校验
- 密码至少 6 位
- 提交调用 `loginByPassword`

展示文案同步替换：

- `邮箱 / 手机号` 改为 `用户名`

### 注册页

表单项：

- 用户名
- 密码
- 重复密码
- Turnstile

行为：

- 用户名按统一正则校验
- 密码至少 6 位
- 重复密码必须一致
- 不再显示验证码输入与发送验证码按钮
- 提交调用新的 `register`

### 本地存储与状态管理

当前前端把登录标识按 `phone` 存储。改造后：

- 本地存储项改为 `username`
- store 中登录态字段同步改名
- 所有依赖 `phone` 展示当前用户身份的页面改为使用 `username` 或 `nickname`

## 错误处理

后端需要提供明确错误：

- 用户名格式不合法
- 用户名已存在
- 用户名不存在
- 密码错误
- Turnstile 未通过
- 服务器未配置 Turnstile

前端需要提供明确提示：

- 用户名仅支持字母、数字、下划线，4-20 位
- 密码至少 6 位
- 两次输入的密码不一致

## 数据迁移与上线策略

由于本次明确不兼容老账号，推荐采用“结构迁移 + 旧数据清理”策略。

### 结构迁移

- 给 `users` 表新增 `username`
- 添加唯一索引
- 保留 `phone` 列，避免当前其他模块直接报错

### 数据清理

上线前清理旧用户数据，避免出现没有 `username` 的旧记录：

- 至少清 `users`
- 若消息、提现、资源、工单等模块仍强依赖用户旧身份，需同步评估是否清理关联数据

本次实现层面不自动帮用户清库，但需要提供明确的线上操作指引。

## 测试设计

### 后端测试

- 用户名合法时可以注册成功
- 非法用户名注册失败
- 重复用户名注册失败
- 正确用户名密码可以登录
- 用户名不存在时登录失败
- 密码错误时登录失败
- 注册成功后返回默认昵称
- `verify` 接口可基于新 token 正常返回

### 前端测试

- 注册页无验证码按钮与验证码输入框
- 用户名格式校验生效
- 两次密码不一致时阻止提交
- 登录页仅保留用户名和密码输入
- 登录成功后本地状态记录的是 `username`

### 集成验证

- 本地注册新用户
- 使用该用户登录
- 修改昵称并刷新后仍能正确显示
- 首个管理员补位逻辑或手动提权逻辑不被破坏

## 实施顺序

1. 调整 `User` 模型与默认昵称生成逻辑
2. 调整 JWT 签发与鉴权解析逻辑
3. 改造认证路由，移除验证码登录注册链路
4. 更新前端接口封装与本地状态字段
5. 改造登录页与注册页
6. 运行后端和前端构建验证
7. 整理线上清库与重建管理员账户步骤

## 风险

- 项目中仍有部分逻辑默认把 `phone` 当用户主标识，需要逐处检查
- 若只改登录页而不改 store / JWT / 鉴权辅助函数，会形成“表单是用户名，内部还是手机号”的伪改造
- 如果线上不清理旧用户数据，可能会出现无 `username` 的记录导致登录与权限判断异常

## 验收标准

- 新用户可使用用户名注册
- 新用户可使用用户名和密码登录
- 邮箱验证码相关入口完全消失
- 默认昵称自动生成且可在资料页修改
- 管理员权限逻辑仍正常
- 前后端不再把 `phone` 当认证主字段使用
